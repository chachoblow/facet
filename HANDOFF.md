## Where things stand

- **Project**: Facet — a markdown editing experience oriented around shared alignment across humans, AI agents, and future-self.
- **V1 target**: Facet for VS Code, a Hybrid live-preview Surface for `.md` files, built on CodeMirror 6.
- **Long-term vision**: Three Surfaces (Facet for VS Code, Facet Review, Facet Studio) sharing a Remark-based markdown core, with the git repo as the Content source of truth.
- **Repo**: <https://github.com/chachoblow/facet> — public, `main` branch.
- **Status**: Monorepo scaffolded (pnpm workspaces + TypeScript project references + esbuild, Node 24 / pnpm 10 pinned). Spikes 1–3 run; all green-lit Plan A. Spike 1 promoted to a permanent test at `packages/core/test/round-trip.test.ts`. **Impl order steps 1–12 are done.** The VS Code extension hosts a `CustomTextEditorProvider` with a CodeMirror 6 webview; edits flow webview → `postMessage` → `WorkspaceEdit` → `TextDocument`, and saves write the buffer's bytes verbatim — the v1 structural mechanism for D5. `priority` is `"option"` so VS Code's default markdown editor stays primary until impl order step 12. `@facet/core` exposes `parse(markdown): Root`, `findInlineMarks(root): InlineMark[]`, `findBlocks(root): Block[]`, `findTables(root): Table[]`, `findFrontmatter(root): Frontmatter | null`, and `findImages(root): Image[]`. `parse.ts` runs `unified + remark-parse + remark-gfm + remark-frontmatter` (configured for `["yaml"]` only, locking D6 — no TOML / `+++` fences); `remark-frontmatter` is now a runtime `dependency`. The webview drives Hybrid live-preview via six parallel StateFields: inline marks (`strong`/`emphasis`/`link`) hide their syntax characters when the cursor is outside the mark's outer range; blocks (`heading`/`blockquote`/`listItem`) get line-level decorations plus marker hiding — headings and list items use a per-block cursor check (`# ` and `- ` reveal when the cursor is on any line of the block), blockquotes use a per-line cursor check (`> ` reveals only on the line the cursor is actually on, sibling lines in the same outer blockquote stay decorated). Selectors are prefixed with `.cm-content` (specificity 0,3,0) to beat CodeMirror 6's auto-generated `.ͼ1 .cm-line`. Unordered list items render as `• ` via a `BulletWidget`, ordered list `1. ` markers stay visible; code blocks are owned end-to-end by `code-marks.ts` (Shiki-based syntax highlighting via `Decoration.mark` per token, both fences collapsed with `Decoration.replace({ block: true })` when the cursor is off the block, the opening fence carrying a `LangBadgeWidget` rendering `block.lang`); tables render as a styled grid (`.facet-table-line` / `.facet-table-header-line` / `.facet-table-alignment-line` line decorations, plus per-cell `.facet-table-cell-{left,center,right}` marks for non-default-aligned columns) with each `|` hidden via `Decoration.replace` when the cursor is on a non-table line, and pipes returning when the cursor is on any line of the table; frontmatter renders as a single collapsed `FrontmatterWidget` ("⌃ Frontmatter") covering the whole `---`…`---` range when the cursor is off any frontmatter line, with `EditorView.atomicRanges` making CM treat the collapsed range as a single jump unit so arrow keys don't drop the cursor at unpredictable offsets across the multi-line replace — clicking the widget dispatches a selection change to land the cursor inside the YAML body; and images render as an `<img>` widget (`ImageWidget`) covering `image.start..image.end` when the cursor's line is outside the image's line range. ## Immediate next action: impl order step 13 — polish (in progress)

The functional surface area is in place (steps 1–12). Step 13 is the catch-all polish pass to close the parked items below before the honest test (live on the team wiki for a week). Polish is broader than previous steps; expect to triage and prioritize before diving in.

**Already cleared in this step:**

- _(TDD-driven, EDH verification still pending)_ Block-level `Decoration.replace({ block: true })` for the frontmatter widget (`apps/vscode/src/webview/frontmatter-marks.ts`) and the table alignment row (`apps/vscode/src/webview/table-marks.ts`) — both ranges now collapse vertical space rather than just being muted by opacity / inline replace.

- _(TDD-driven plus EDH-verified)_ Visual depth + per-line reveal for nested blockquotes (`apps/vscode/src/webview/block-marks.ts`, `apps/vscode/src/provider.ts`). Each level emits a depth-encoded line class (`.facet-blockquote-line-{1..6}`, deeper levels fall back to depth-6 styling); the CSS uses a stacked `box-shadow` chain with alternating bar/gap inset shadows so a depth-N line renders as N vertical bars at ch-aligned positions. Cursor reveal flips from per-block to per-line for blockquotes only — clicking into `> > inner` reveals only that line's `> >`, sibling lines in the same outer blockquote stay decorated. Selectors are prefixed with `.cm-content` (specificity 0,3,0) to beat CodeMirror 6's auto-generated `.ͼ1 .cm-line` rules. Two cosmetic shifts left as known-minor (see Pending / parked): non-canonical `>>` (no space between markers) shifts ~1ch on cursor toggle; canonical `> >` shifts a fraction of a char from the CSS `ch` unit measuring `0` advance width rather than `>`.

**Remaining, roughly ordered by user-facing impact:**

1. **ESM webview refactor — closes the bundle-size + lazy-load items together.** Today the webview is **4.4 MB** because esbuild's IIFE format doesn't support code-splitting; the dynamic `import("mermaid")` and Shiki's grammars all fold into the static bundle. The fix is to retarget `apps/vscode/esbuild.config.mjs`'s webview build from IIFE to ESM with chunk loading, then update the webview HTML to load the entry as `<script type="module">`. This is the bigger move than typical polish — flag the size when scheduling. User-facing impact: faster first-paint on opening a `.md` file, especially on the team-wiki test.
2. **Cursor edge cases on decorated lines — closes the step-5 arrow-up/down item and the step-11 selection-chrome item.** Pressing Down across a decorated line lands at a surprising source offset because CM tracks pixel-x as the goal column, and decorated lines have less visible width than their source. The selection-chrome flicker (parked in step 11) shares the root cause: reveal-on-cursor StateFields rebuild on selection-only transactions, invalidating selection/cursor positions mid-paint. A coherent fix is probably either decoupling reveal-on-cursor rebuilds from selection-only transactions (so the selection layer doesn't repaint mid-drag), or layering a custom keymap that preserves source offset when source and target lines have the same structural prefix. Don't fix one without the other — they share machinery.
3. **`localResourceRoots` for files outside any workspace (step-8 follow-up).** Files opened with no workspace are limited to their own directory tree; cross-directory `../shared/foo.png` references resolve syntactically but the fetch is blocked. Fix on the extension side: derive additional roots from each image's resolved path before mounting `webview.options`. Address only if it bites real Authors.
4. **D8 revisit at the polish checkpoint** (line-based reveal vs. overlay-layer rendering for complex blocks). Steps 7, 8, and 10 all confirmed line-based reveal fits cleanly so the overlay pivot stayed parked. Step 13 is the final D8 checkpoint per `docs/facet-decisions.md`. Re-read the decision before reaching for an overlay layer for a specific block type.
5. **EDH verification of the still-pending TDD-driven changes (frontmatter widget block-collapse, table alignment row block-collapse, cell-align mark scoping inside pipes).** Decoration-model changes can pass unit tests while having layout surprises — the nested-blockquote slice was the cautionary tale (passed all tests, but visual depth wasn't communicated and CM6's `.ͼ1 .cm-line` specificity quietly nullified our padding). Likely candidates to eyeball: the frontmatter widget now sits inside CM's block-widget container with its existing `display: inline-block` styling; the alignment row collapse interacts with `.facet-table-line` borders; cell-align marks should no longer drag the leading `|` when pipes are visible.

Lower-priority items that may or may not surface as polish (defer if not user-visible): cross-file anchor cursor displacement (step-9 cosmetic — `revealRange` + cursor selection feels off in Facet because the webview owns scrolling).

Out of scope for v1 polish: editing-mode toggle, properties panel for frontmatter, paste/drag-drop image, slash commands, custom toolbar, true WYSIWYG (all explicit non-goals — see `docs/facet-spec-v1.md`). Don't let polish creep into feature work.

Guardrails worth re-reading first: D5 (round-trip — none of these polish items touch the save path, but the temptation to "normalize while we're already in there" is real), D6 (no custom markdown syntax — same), D8 (block rendering model — final checkpoint, capture the outcome in `docs/facet-decisions.md`), and the v1 non-goals list in the spec.

Decide before starting: with the nested-blockquote slice EDH-verified and the other three cosmetic items still TDD-only, the remaining work is the "13a — bundle + cursor (high-impact)" half plus the EDH verification chore for those three. The bundle refactor and cursor work share no machinery, so they can ship as separate sprints if helpful. The cautionary lesson from this slice: tests can lock in pure logic but won't catch CSS-specificity or visual-design issues — keep EDH verification in the loop for any decoration-model change.

## Guardrails for the next session

A few things easy to forget under time pressure:

- **Round-trip fidelity (D5).** The v1 mechanism is structural: CodeMirror owns the text buffer and saves write its bytes verbatim — `remark-stringify` is not called on the save path. Don't add a save-path serializer. Don't let CodeMirror, Remark config, or any plugin silently normalize list markers, reflow paragraphs, or rewrite link forms. The round-trip test at `packages/core/test/round-trip.test.ts` locks Remark behavior — keep it green and don't disable it. `remark-stringify` is a `devDependency` of `@facet/core` for the test only; don't promote it to `dependencies` or import it from `src/`.
- **No custom markdown syntax (D6).** CommonMark + GFM only. No wiki-style `[[links]]`, custom callouts, or embed syntax.
- **Use the canonical terms** from `UBIQUITOUS_LANGUAGE.md`. Especially **Surface** vs **Facet** and **Thread** vs **Comment**.
- **Plan B is real (D3).** If CodeMirror Hybrid live-preview turns into disproportionate custom-plugin work, switching Facet for VS Code to **Milkdown with a source-mode toggle** is a legitimate move, not a failure. Keep `packages/core/` library-agnostic so the swap stays feasible.
