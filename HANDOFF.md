## Immediate next action: impl order step 11 — VS Code theme integration

Theme integration. The v1 spec (`docs/facet-spec-v1.md`) calls for the Facet Surface to render correctly under VS Code's three theme kinds — light, dark, and high-contrast — without per-Author configuration. The current state is dark-leaning by accident: Shiki tokens use `github-dark` (step 7), Mermaid is initialized with `theme: "dark"` (step 10), and most CSS uses VS Code theme variables (e.g., `var(--vscode-editor-foreground)`) but a few colors are hardcoded.

Three places need theme-aware behavior:

1. **Shiki theme selection** (`apps/vscode/src/webview/code-marks.ts`). The highlighter currently bundles only `github-dark`. The shape is to bundle `github-dark` + `github-light` (and possibly a high-contrast theme), pick at view-creation time based on the active VS Code theme kind, and re-pick if the Author switches themes mid-session. VS Code exposes this via `vscode.window.activeColorTheme.kind` (`Light` / `Dark` / `HighContrast` / `HighContrastLight`); the extension can ship the kind in the `update` message alongside `text` and `baseUri`. The existing `highlighterReady` await stays — pick the theme inside `createView`. A single re-pick on theme change can be handled by tearing down the StateField extension and rebuilding the view, or by making the chosen theme reactive (a `Compartment`).
2. **Mermaid theme selection** (`apps/vscode/src/webview/mermaid-marks.ts`). The `initialize({ theme: "dark" })` call needs to read VS Code's theme kind. Mermaid accepts `default` / `dark` / `neutral` / `forest` — map dark→`"dark"`, light→`"default"`, high-contrast→`"neutral"` (or pick what looks right by eye). Mermaid's initialize is global, so re-initializing on theme change is fine; existing rendered widgets will need their `eq()` to bust on theme change so they re-render. Either factor the theme into the widget's `eq()` key, or wipe the lazy `mermaidPromise` on theme change and force re-render via a CodeMirror extension swap. (The latter is cleaner — keeps `eq()` simple.)
3. **Provider CSS pass** (`apps/vscode/src/provider.ts`). Audit the CSS in `getHtml()` — replace any remaining hardcoded colors with `var(--vscode-...)` fallbacks. Particular attention to `.facet-mermaid-error` (currently uses a hardcoded `#f48771` fallback for `--vscode-errorForeground`), the `rgba(127,127,127,...)` neutrals scattered throughout, and anything that assumes a dark background. The `.facet-bullet` uses `--vscode-textLink-foreground` which adapts; the heading line classes use VS Code-relative font sizes which adapt. Most of this is already correct — the audit is to find what isn't.

Out of scope this step: full custom theme picker for Facet, Mermaid PNG export (D-list open question), accessibility audit beyond what high-contrast theme support naturally gives.

Guardrails worth re-reading first: D5 (round-trip — theme changes don't touch the buffer), D6 (theme behavior is presentation, not markdown semantics), and the v1 non-goal "no Author preferences UI" (themes follow VS Code's setting, not a Facet-specific setting).

Step 10 follow-ups parked (see Pending / parked below): the webview bundle grew from 1.5 MB to **4.4 MB** because esbuild's IIFE format folds the dynamic `import("mermaid")` into the static bundle (lazy-load contract is broken architecturally) — folded with the step-7 Shiki bundle item into step 13 polish. The proper fix is the ESM-webview refactor that closes both items together; doing it during step 11 would be a tangent.

Earlier follow-ups still safe to defer (see Pending / parked below): nested blockquotes (`>>` / `>>>`) collapse visually to one quote level (step 4); arrow-up/down across decorated lines lands at a surprising source offset because CM tracks pixel-x (step 5, defer to step 13 with a unified custom keymap); cross-file `.md` links open in VS Code's default markdown editor because Facet is `priority="option"` (step 9, resolves at step 12).

Two cosmetic items deferred from step 5 (acceptable for v1, fold into polish): the alignment row is muted via opacity rather than collapsed (same `Decoration.replace({ block: true })` mechanism step 7 used for code fences would resolve it), and cell-align marks use `display: inline-block` so the leading `|` of cells 2+ inherits the alignment when the cursor is inside the table. Both are CSS-only and easy to retune later.

Two parked from step 6 (frontmatter, fold into step 13 polish): the collapsed `FrontmatterWidget` uses inline `Decoration.replace`, which leaves the source line breaks consuming vertical space (same fix as the table alignment row — `Decoration.replace({ block: true })` over the whole range). And `atomicRanges` resolves the worst arrow-key drift but doesn't deliver the Obsidian "click into a layer" feel — that's the D8 revisit territory, not a one-off frontmatter fix.

One parked from step 8 (image rendering, fold into step 13 polish): `localResourceRoots` is widened to the workspace folder (or the doc's directory for files outside any workspace). For files in a workspace this covers most images, including `../../shared/logo.png`. Files opened outside any workspace are limited to their own directory tree — a reference like `![](../shared/foo.png)` will resolve syntactically (the URL constructor is happy) but the image fetch will be blocked. Acceptable for v1; if it bites real Authors, the fix is to add additional roots derived from each image's resolved path on the extension side.

**D8 outcome at the step-10 checkpoint** (recorded in `docs/facet-decisions.md`): line-based reveal fits Mermaid cleanly. The next D8 checkpoint moves to step 13 polish.

## Guardrails for the next session

A few things easy to forget under time pressure:

- **Round-trip fidelity (D5).** The v1 mechanism is structural: CodeMirror owns the text buffer and saves write its bytes verbatim — `remark-stringify` is not called on the save path. Don't add a save-path serializer. Don't let CodeMirror, Remark config, or any plugin silently normalize list markers, reflow paragraphs, or rewrite link forms. The round-trip test at `packages/core/test/round-trip.test.ts` locks Remark behavior — keep it green and don't disable it. `remark-stringify` is a `devDependency` of `@facet/core` for the test only; don't promote it to `dependencies` or import it from `src/`.
- **No custom markdown syntax (D6).** CommonMark + GFM only. No wiki-style `[[links]]`, custom callouts, or embed syntax.
- **Use the canonical terms** from `UBIQUITOUS_LANGUAGE.md`. Especially **Surface** vs **Facet** and **Thread** vs **Comment**.
- **Plan B is real (D3).** If CodeMirror Hybrid live-preview turns into disproportionate custom-plugin work, switching Facet for VS Code to **Milkdown with a source-mode toggle** is a legitimate move, not a failure. Keep `packages/core/` library-agnostic so the swap stays feasible.
